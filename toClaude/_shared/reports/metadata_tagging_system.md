# onTong 메타데이터 태깅 시스템

> 발표용 리포트 | 작성일: 2026-03-26

## 1. 왜 메타데이터 태깅인가?

### 기존 문제: 단순 텍스트 검색의 한계

제조 SCM 현장의 지식 문서는 수백~수천 건에 달하며, 내용이 유사한 문서가 많습니다.
벡터 유사도 검색(RAG)만으로는 다음 상황에서 정확도가 떨어집니다:

| 상황 | 벡터 검색 결과 | 문제 |
|------|----------------|------|
| "DG320 에러 해결법 알려줘" | DG320과 유사한 단어가 포함된 10개 문서 반환 | 정작 DG320 장애 대응 문서가 3번째 |
| "SCM 주문 처리 규칙" | 주문 관련 모든 도메인 문서 반환 | QC, 물류 문서까지 섞임 |
| "재고관리 프로세스에서 FIFO 적용 사례" | FIFO 키워드 포함 문서 반환 | 재고관리가 아닌 다른 프로세스 문서 포함 |

### 해결: 구조화된 메타데이터 + 벡터 검색 = Hybrid Search

```
[사용자 질문] → AI가 질문 분석 → 메타데이터 필터 자동 추론
                                         ↓
                              ChromaDB where 필터 적용
                                         ↓
                              필터링된 범위 내에서 벡터 검색
                                         ↓
                              정확도 높은 Top-K 문서 반환
```

## 2. 메타데이터 구조 설계

### YAML Frontmatter 기반

모든 `.md` 문서 상단에 YAML 형식으로 메타데이터를 기록합니다:

```yaml
---
domain: SCM
process: 주문처리
tags:
  - 주문
  - 재고
  - FIFO
error_codes:
  - DG320
author: admin
created: "2026-03-20"
---
```

### 필드 설계 근거

| 필드 | 역할 | 검색 활용 |
|------|------|-----------|
| **domain** | 업무 영역 분류 (SCM, QC, 생산, 물류 등) | 1차 필터 — 도메인 범위 한정 |
| **process** | 세부 프로세스 (주문처리, 입고, 검수 등) | 2차 필터 — 프로세스 범위 한정 |
| **tags** | 자유 키워드 태그 | 벡터 검색 보완 — 의미 기반 매칭 |
| **error_codes** | 에러/장애 코드 | 정확 매칭 — 장애 대응 시 즉시 검색 |
| **author** | 작성자 | 책임 추적, 전문가 식별 |
| **created** | 작성일 | 시간 기반 필터링 |

## 3. 기술 아키텍처

### 전체 흐름

```
[문서 작성/수정]
    ↓
[YAML Frontmatter 파싱] ← local_fs.py: _parse_frontmatter()
    ↓
[ChromaDB 인덱싱] ← wiki_indexer.py: 파이프 구분자 메타데이터
    ↓
[AI Copilot 질문]
    ↓
[RAG Agent] → metadata_filter → ChromaDB query_with_filter(where=...)
    ↓
[필터링된 벡터 검색 결과] → LLM 답변 생성
```

### ChromaDB 메타데이터 저장 방식

ChromaDB의 `$contains` 쿼리를 활용하기 위해 **파이프 구분자** 형식으로 저장:

```python
# 저장 시
metadata = {
    "domain": "SCM",
    "process": "주문처리",
    "tags": "|주문|재고|FIFO|",        # 파이프 구분자
    "error_codes": "|DG320|",
}

# 검색 시
where = {"tags": {"$contains": "|FIFO|"}}
```

### LLM Auto-Tag

사용자가 직접 태그를 지정하지 않아도 LLM이 문서 내용을 분석해 메타데이터를 추천합니다:

```
[문서 본문 (최대 3000자)]
    ↓
[LLM (GPT-4o-mini)] → JSON 응답
    ↓
{
  "domain": "SCM",
  "process": "주문처리",
  "tags": ["주문", "재고", "FIFO", "만료처리"],
  "error_codes": ["DG320"],
  "confidence": 0.85
}
    ↓
[사용자 확인] → 수락/거절/수정
```

## 4. 사용자 경험 (UX)

### MetadataTagBar

에디터 상단에 위치한 메타데이터 입력 바:

- **Domain / Process**: 드롭다운 선택 (기본 옵션 + API에서 가져온 기존 값)
- **Tags**: Badge 기반 태그 입력 + 자동 완성
- **Auto-Tag 버튼**: LLM 추천 → 점선 Badge로 미리보기 → 수락/거절

### 핵심 UX 원칙

1. **제로 설정**: 태그를 모르는 사용자도 Auto-Tag로 시작 가능
2. **점진적 학습**: 기존 태그가 자동 완성으로 제안되어 일관성 유지
3. **비파괴적**: 메타데이터는 문서 본문과 분리 (frontmatter), 기존 워크플로 방해 없음

## 5. 고도화 로드맵

### Phase 2에서 계획된 개선

| # | 기능 | 효과 | 우선순위 |
|---|------|------|----------|
| 1 | **저장 시 자동 태깅** | 태그 커버리지 100% — 미태깅 문서 제로 | 높음 |
| 2 | **에러코드 자동 추출** | LLM 호출 없이 즉시 처리, 장애 대응 속도 향상 | 높음 |
| 3 | **태그 정규화** | "캐시"/"캐쉬"/"cache" 통합 → 검색 누락 방지 | 중간 |
| 4 | **태그 기반 사이드바** | Domain > Process > Tags 계층 탐색 | 중간 |
| 5 | **AI 질문 시 자동 필터 추론** | 사용자 개입 없이 RAG 정확도 향상 | 높음 |
| 6 | **미태깅 문서 대시보드** | 태그 관리 현황 가시화, 일괄 처리 | 낮음 |

### 기대 효과 (정량)

| 지표 | 현재 (벡터만) | 메타데이터 적용 후 (예상) |
|------|---------------|--------------------------|
| RAG Top-3 정확도 | ~60% | **85%+** |
| 장애코드 검색 응답 시간 | 벡터 유사도 3-5초 | **정확 매칭 <1초** |
| 문서 태그 커버리지 | 0% (수동) | **100%** (자동 태깅) |
| 도메인 필터링 정밀도 | N/A | **95%+** |

## 6. 경쟁 우위

### vs. 일반 RAG 시스템

| 비교 항목 | 일반 RAG | onTong |
|-----------|----------|--------|
| 검색 방식 | 벡터 유사도만 | 메타데이터 필터 + 벡터 Hybrid |
| 태그 관리 | 없음 또는 수동 | LLM Auto-Tag + 자동 완성 |
| 도메인 인식 | 없음 | Domain/Process 기반 범위 한정 |
| 장애 대응 | 키워드 검색 | 에러코드 정확 매칭 |
| 메타데이터 저장 | 별도 DB | 문서 내 Frontmatter (이식성 높음) |

### 핵심 차별점

1. **문서 자체에 메타데이터 내장** — 별도 DB 불필요, git으로 버전 관리 가능
2. **AI 보조 태깅** — 사람이 일일이 분류하지 않아도 LLM이 자동 추천
3. **제조 SCM 특화 분류 체계** — 범용 태그가 아닌 도메인/프로세스/에러코드 3축 분류

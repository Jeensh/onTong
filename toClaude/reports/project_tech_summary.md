# onTong - 사내 위키 AI 에이전트 기술 발표 자료

## 1. 프로젝트 개요

**onTong**은 사내 위키 문서를 AI 에이전트가 이해하고 검색/생성/수정할 수 있는 **RAG 기반 지식 관리 플랫폼**입니다.

### 핵심 가치
- **질문하면 답한다**: 사내 위키 문서 기반 RAG로 정확한 답변 제공
- **문서를 만들고 고친다**: AI가 초안을 작성하고, 사람이 승인하는 Human-in-the-Loop
- **실시간으로 보여준다**: SSE 스트리밍으로 사고 과정까지 투명하게 표시

## 4. 핵심 기술 전략

### 4.1 RAG 파이프라인 (5-Stage)

단순 검색-답변이 아닌, **5단계 인지 파이프라인**을 설계했습니다.

```
① Query Augmentation (쿼리 증강)
   → 멀티턴 대화에서 "그거"/"그 문서" 같은 대명사를 실제 맥락으로 치환
   → 예: "그거 수정해줘" → "직원정보-마케팅DX그룹.md 문서를 수정해줘"

② Vector Search (벡터 검색)
   → ChromaDB에서 코사인 유사도 기반 Top-8 검색
   → relevance = max(0, 1 - distance), 임계값 0.55 이하 필터링

③ Clarity Check (명확성 검증)
   → LLM이 질문의 모호성을 판단
   → 모호하면 역질문: "어떤 팀의 직원 정보를 원하시나요?"

④ Cognitive Reflection (인지적 성찰) ← 핵심 차별점
   → 숨겨진 내부 추론 단계 (사용자에게 노출 안 됨)
   → "사용자가 진짜 원하는 것은?", "감정 상태는?", "도메인 맥락은?"
   → 초안 → 자기비판 → 개선된 최종 답변

⑤ Final Answer (최종 답변 스트리밍)
   → 공감 → 결론 → 근거 → 후속 액션 (민토 피라미드)
   → 출처 인용 필수
```

**왜 이렇게 설계했나?**
- 단순 RAG는 질문이 모호하면 엉뚱한 답변을 함
- Clarity Check로 모호함을 사전 차단
- Cognitive Reflection으로 답변 품질을 자체 검증
- 결과: 질문 1개에 LLM 호출 3~4회지만, 답변 정확도와 자연스러움이 크게 향상

### 4.2 문서 청킹 전략 (Heading-Based + Overlap)

```
원본 문서:
  # 제목
  ## 섹션 A (200 tokens)
  ## 섹션 B (800 tokens) ← MAX_CHUNK(500) 초과
  ## 섹션 C (100 tokens)

청킹 결과:
  chunk_0: "# 제목" (전체)
  chunk_1: "## 섹션 A" (전체, 200 tokens)
  chunk_2_0: "## 섹션 B" 전반부 (500 tokens)
  chunk_2_1: "## 섹션 B" 후반부 (350 tokens, 50 tokens 오버랩)
  chunk_3: "## 섹션 C" (전체, 100 tokens)
```

- **마크다운 헤딩 기반 분리**: 문서의 의미 단위를 보존
- **500 토큰 제한 + 50 토큰 오버랩**: 섹션 경계에서 정보 손실 방지
- **짧은 문서 보강**: 150 토큰 미만 문서는 파일 경로를 접두어로 추가하여 임베딩 품질 향상
  - 예: `[문서: 직원정보 > 마케팅DX그룹] 이승현 수석...`
- **토큰 추정**: `len(text) // 3` (한영 혼합 텍스트에 최적화된 근사치)

### 4.3 메타데이터 시스템 (YAML Frontmatter)

모든 위키 문서에 YAML 프론트매터로 구조화된 메타데이터를 부여합니다.

```yaml
---
domain: HR           # 도메인 (HR, IT, SCM 등)
process: 인사관리      # 업무 프로세스
tags:                 # 키워드 태그
  - 인사
  - 직원정보
error_codes:          # 관련 에러 코드
  - ERR-001
created: '2026-03-20T09:00:00Z'
updated: '2026-03-27T06:32:43Z'
created_by: 관리자
updated_by: 개발자
---
```

**ChromaDB 저장 방식**: 파이프 구분자(`|태그1|태그2|`)로 플래팅하여 `$contains` 쿼리 지원

**자동 태그 제안**: LLM이 문서 본문을 분석하여 적절한 domain/process/tags를 추천

### 4.4 Human-in-the-Loop 승인 플로우

AI가 문서를 직접 수정하지 않고, **사람의 승인을 거치는 안전장치**를 두었습니다.

```
사용자: "마케팅DX그룹 직원 정보에 김태헌 OJT 완료로 수정해줘"
    ↓
AI: 의도 감지 (Edit Intent)
    ↓
AI: 대상 문서 탐색 (Vector Search or 첨부 파일)
    ↓
AI: LLM으로 수정본 생성 + diff 미리보기
    ↓
[승인 요청 UI 표시]
  ├→ ✅ 승인 → 파일 저장 + ChromaDB 재인덱싱 + 사이드바 갱신 + Diff 뷰 표시
  └→ ❌ 거절 → 아무 변경 없음
```

**액션 타입**: `wiki_write` (신규 생성), `wiki_edit` (수정), `wiki_delete` (삭제)

### 4.5 실시간 SSE 스트리밍

응답을 토큰 단위로 스트리밍하며, **사고 과정도 실시간 표시**합니다.

```
event: routing          → "WIKI_QA 에이전트로 라우팅"
event: thinking_step    → "관련 문서 검색 중..."
event: thinking_step    → "문서 8건 검색 완료 (최고 관련도 95%)"
event: thinking_step    → "답변 품질 검증 중..."
event: sources          → [{doc: "직원정보.md", relevance: 0.95}]
event: content_delta    → "마" → "케팅" → "DX" → "그룹의"...
event: done             → {usage: {input_tokens: 1200, output_tokens: 350}}
```

**기술 결정**:
- Next.js 프록시는 SSE 버퍼링 문제가 있어 **백엔드 직접 연결**
- `X-Accel-Buffering: no` 헤더로 nginx 버퍼링 비활성화
- Fetch API + ReadableStream으로 수동 SSE 파싱 (EventSource 대비 POST 지원)

### 4.6 파일 첨부 (강제 참조)

사용자가 특정 파일을 **📎 첨부**하면, 벡터 검색을 건너뛰고 해당 파일을 직접 참조합니다.

```
일반 모드: 질문 → 벡터 검색 → Top-K 결과 → LLM 답변
첨부 모드: 질문 + 📎파일 → 파일 내용 직접 로드 → LLM 답변 (검색 스킵)
```

**활용 사례**: "이 문서에서 김태헌 정보 수정해줘" → 정확한 파일 타겟팅

### 4.7 Diff 뷰 (변경사항 시각화)

문서 수정 승인 시, **어디가 바뀌었는지** 워크스페이스에서 시각적으로 확인할 수 있습니다.

- 승인 전 내용을 캡처 → 승인 후 새 내용과 비교
- LCS 기반 라인 diff 알고리즘
- 삭제 라인: 빨간 배경 + 취소선
- 추가 라인: 초록 배경
- 라인 번호 + ±표시로 변경 위치 한눈에 파악
- "확인 완료" 클릭 시 일반 에디터로 복귀

### 4.8 벡터DB 동기화 파이프라인

문서의 생성/수정/삭제가 즉시 ChromaDB에 반영됩니다.

| 이벤트 | 동작 |
|--------|------|
| 문서 저장 (PUT) | 기존 청크 삭제(`delete_where`) → 새로 인덱싱 |
| 문서 삭제 (DELETE) | 메타데이터 필터로 해당 파일 청크 전체 삭제 |
| 서버 시작 | 전체 컬렉션 초기화 → 모든 파일 리인덱싱 |

## 6. 에이전트 라우팅 전략

```
사용자 입력
    ↓
┌─ Tier 1: 키워드/정규식 매칭 (27개 규칙) ─┐
│  "문서 찾아" → WIKI_QA                     │
│  "시뮬레이션" → SIMULATION                  │
│  "원인 추적" → DEBUG_TRACE                  │
└──────────────────────────────────────────┘
    ↓ (매칭 실패 시)
┌─ Tier 2: LLM 분류 ─────────────────────────┐
│  Claude가 입력을 분석하여 에이전트 선택       │
│  confidence 점수와 함께 반환                  │
└─────────────────────────────────────────────┘
```

**설계 이유**: 키워드 매칭은 빠르고 저비용, LLM 분류는 정확하지만 느림 → 2-Tier로 최적화

## 7. 보안 및 확장성 설계

### 인증 추상화
- Factory Pattern으로 인증 프로바이더 교체 가능
- 현재: `noop` (개발 모드), 향후: OIDC / LDAP / SAML 전환 가능

### 세션 관리
- 인메모리 세션 저장소 (경량, 서버 재시작 시 초기화)
- 세션별 대화 이력 + 승인 대기 액션 관리

### LLM 추상화
- LiteLLM으로 모델 교체 자유 (Claude ↔ GPT ↔ Ollama 등)
- `.env` 설정만 변경하면 다른 모델로 전환

## 8. 주요 기술적 도전과 해결

| 도전 | 해결 |
|------|------|
| 한영 혼합 텍스트의 토큰 추정 | `len(text) // 3` 근사식 (실측 기반) |
| 모호한 질문에 대한 오답 | Clarity Check 단계 추가, 역질문 유도 |
| 답변 품질 불균일 | Cognitive Reflection (숨겨진 자기비판) |
| ChromaDB 스테일 데이터 | 메타데이터 기반 삭제 + 서버 시작 시 전체 리인덱스 |
| 첨부 파일 무시하고 벡터 검색 | attached_files 우선 처리, 벡터 검색 스킵 |
| SSE 버퍼링 (Next.js 프록시) | 백엔드 직접 연결로 우회 |
| 승인 후 사이드바 미갱신 | Zustand `treeVersion` 카운터로 반응형 갱신 |
| 수정 후 변경점 불투명 | LCS diff 알고리즘으로 인라인 Diff 뷰 구현 |

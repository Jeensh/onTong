# 롤백 절차 가이드

## 언제 롤백하는가

배포 후 다음 조건 중 **하나라도** 충족되면 롤백을 고려합니다:

- 에러율이 배포 전 대비 **2배 이상** 증가
- P95 응답시간이 **500ms 이상** 증가
- 핵심 기능(주문, 결제, 로그인) 실패율 **1% 초과**
- Canary 배포 단계에서 비정상 지표 감지

> 판단이 어려우면 **롤백 우선**. 원인 분석은 롤백 후에 합니다.

## 환경별 롤백 방법

### Kubernetes (ArgoCD)

```bash
# 1. 현재 상태 확인
argocd app get <app-name>

# 2. 이전 리비전으로 롤백
argocd app rollback <app-name> <revision>

# 3. 동기화 상태 확인
argocd app wait <app-name> --health
```

### Kubernetes (kubectl 직접)

```bash
# 배포 히스토리 확인
kubectl rollout history deployment/<name> -n <namespace>

# 이전 버전으로 롤백
kubectl rollout undo deployment/<name> -n <namespace>

# 특정 리비전으로 롤백
kubectl rollout undo deployment/<name> --to-revision=<N>

# 롤백 상태 확인
kubectl rollout status deployment/<name> -n <namespace>
```

### DB 마이그레이션 롤백

```bash
# Alembic (Python)
alembic downgrade -1

# Flyway (Java)
flyway undo
```

> DB 롤백은 **데이터 손실 가능성**이 있으므로 반드시 DBA와 협의합니다.

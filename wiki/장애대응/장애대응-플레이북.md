# 장애대응 플레이북

## 목적

장애 발생 시 **누가, 무엇을, 언제** 해야 하는지 명확히 정의합니다. 이 문서는 [[장애등급-분류기준]]과 함께 사용합니다.

## 대응 역할

| 역할 | 담당 | 책임 |
|------|------|------|
| **Incident Commander (IC)** | 당직 SRE | 전체 조율, 의사결정 |
| **Ops Lead** | 인프라팀 | 시스템 복구 실행 |
| **Comms Lead** | 서비스기획팀 | 내/외부 커뮤니케이션 |
| **Subject Matter Expert** | 해당 서비스 개발자 | 원인 분석, 핫픽스 |

## P1 장애 대응 타임라인

### T+0분: 감지 및 선언
- 모니터링 알림 또는 사용자 제보로 인지
- 슬랙 `#incident-war-room` 채널에 장애 선언
- IC 자동 지정 (당직 SRE 로테이션)

### T+5분: 초기 진단
- 영향 범위 파악 (서비스, 사용자 수, 매출 영향)
- [[장애등급-분류기준]]에 따라 등급 확정
- 상태 페이지 업데이트

### T+15분: 원인 분석
- 최근 배포 이력 확인 → 배포가 원인이면 즉시 [[롤백-절차-가이드]] 실행
- 인프라 지표 확인: CPU, 메모리, 디스크, 네트워크
- DB 슬로우 쿼리, 커넥션 풀 상태 확인

### T+30분: 복구 또는 우회
- 근본 원인 수정 or 임시 우회 조치
- 부분 복구 시 상태 페이지 업데이트
- 30분마다 경영진 브리핑 (P1)

### T+완료: 복구 확인
- 서비스 정상 확인 (헬스체크 + 실사용자 테스트)
- 장애 종료 선언
- 48시간 내 [[사후분석-템플릿]] 작성

## 커뮤니케이션 템플릿

### 장애 선언
```
🚨 [P1 장애] {서비스명} 서비스 장애 발생
- 시각: YYYY-MM-DD HH:MM
- 영향: {영향 범위}
- IC: @{담당자}
- 상태: 원인 분석 중
```

### 복구 완료
```
✅ [P1 복구] {서비스명} 서비스 정상화
- 복구 시각: YYYY-MM-DD HH:MM
- 원인: {간단 원인}
- 사후 분석: {일정}
```
